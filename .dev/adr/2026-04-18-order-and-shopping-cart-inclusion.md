# ADR: 프로젝트에 Order 기능과 Shopping Cart를 함께 이번 릴리스에 포함할지 검토

**날짜**: 2026-04-18  
**상태**: Decided  
**의결 방식**: Council Debate (4-panel structured deliberation)

---

## 결정

**결정**: Order 기능과 Shopping Cart를 **이번 릴리스에 함께 포함**한다. 

단, 다음 조건에서만 착수하며, MVP 스코프는 카트의 기본 4개 작업(추가/삭제/수량 변경/총액 조회)으로 엄격히 제한한다.

---

## 컨텍스트

현 프로젝트는 키오스크 기반 주문 시스템으로, order 기능(메뉴 선택, 결제)의 구현이 진행 중이다. 질문: "이 order 기능에 shopping cart를 함께 포함할 것인가, 아니면 order만 먼저 출시하고 추후 반복으로 cart를 추가할 것인가?"

이 결정은 다음에 영향을 미친다:
- **제품 완성도**: 카트 없이 order만 지원하면 "실수 수정 불가, 최종 확인 불가"라는 UX 제약이 생김
- **개발 비용**: 순차 출시 시 order API 재설계 비용 vs 동시 설계 시 선행 결정 비용의 트레이드오프
- **기술 부담**: 카트 상태관리, 가격 스냅샷, 도메인 규칙 충돌 등이 실제 복잡도인지 과대추정인지

## 토론 로그

### Phase 1 주요 이견 포인트

**Tension 1: 카트 필요성 vs 가정의 편향**
- ux-specialist, product-owner, tech-architect: "카트는 현대 커머스 UX 표준, 필수"
- devils-advocate: "표준은 가정일 뿐. order만 출시하여 실제 이탈 데이터로 검증해야 함"

**Tension 2: 기술 복잡도의 규모**
- devils-advocate: "상태관리, 세션 병합, 동시성, 가격 스냅샷으로 대규모 리팩토링. 2~3 스프린트"
- tech-architect: "Order 애그리게이트가 이미 카트 개념을 내재. use case 분해만으로 충분. 2~3일"

**Tension 3: 비용 계산의 기준**
- product-owner: "순차 출시 시 order API 재설계 비용이 2~3배. 동시 설계가 경제적"
- devils-advocate: "카트 개발 자체가 2~3 스프린트. 다른 기능(할인, 쿠폰, 배송)에 기회비용 발생"

### Phase 2 반박 및 응답

#### devils-advocate → tech-architect: "카트 기술 부담 과소평가"

**반박 핵심**:
- 카트는 단순 CRUD가 아니라 "가격/재고의 시점 스냅샷 + 세션 병합 + 만료 정책"을 요구하는 상태 머신
- 엔터프라이즈 이커머스의 숨은 복잡도(재고 락킹, 멀티 디바이스 동기화)를 과소평가하면 안 됨

**응답**: shift: yes (부분)
- tech-architect가 코드 기반 반박(`order.py:33-83`)으로 devils-advocate의 전제를 검증
- 이 프로젝트는 엔터프라이즈 이커머스가 아니라 **단일 프로세스 키오스크**
- `InMemoryOrderRepository`에서 세션 병합·멀티 디바이스는 **구조적으로 발생하지 않음**
- devils-advocate가 "대규모 리팩토링" 주장 철회 + "2~3 스프린트" 추정 철회

#### ux-specialist → devils-advocate: "편향 프레임과 베이스라인 독단"

**반박 핵심**:
- devils-advocate의 "카트 없이 order만으로도 수익화 가능"은 베이스라인 설정 오류
- 키오스크 UX에서 "실수 되돌릴 지점 없음"의 이탈 비용이 카트 개발 비용보다 높음

**응답**: shift: yes (부분)
- ux-specialist도 devils-advocate의 "Buy Now 전환율이 더 높다" 주장을 수용
- 단 "비즈니스 모델 검증"을 선행 조건으로 추가: 다품목 구매가 핵심 플로우인가?
- 만약 단일상품 중심이면 order + Buy Now 단독 출시도 가능

#### product-owner → devils-advocate: "MVP 경계 및 순차 비용"

**반박 핵심**:
- product-owner의 "4개 작업 엄격" MVP 제약이 기준이 되어야 함
- "카트라고 부르려면 최소 5개 작업(추가/삭제/수량/총액 + 체크아웃)" vs devils-advocate의 기회비용 논증은 별개 논의

**응답**: shift: yes (조화)
- devils-advocate가 "ADR에 범위 락을 명시하면 스코프 크리프 통제 가능"을 인정
- product-owner가 devils-advocate의 제안("비즈니스 모델 확인을 선행 안건으로")을 ADR에 반영

## 트레이드오프

| 선택지 | 장점 | 단점 |
|--------|------|------|
| **Option A: Cart + Order 동시 출시 (결정)** | · 제품 완성도 높음 (실수 수정 가능) · 후속 재작업 비용 없음 · LineItem 값 객체 공유로 기술 부담 경감 · 동시 설계 시 인터페이스 안정성 확보 | · 초기 복잡도: 선행 결정 3개(매핑, 중복 규칙, 가격 정책) 필요 · 도메인 규칙 변경 필요(`Order.add_item` 중복 금지 → 수량 누적) · 1 스프린트 일정 투자 |
| **Option B: Order 단독 출시 (거절)** | · 빠른 초기 출시(1~2일) · 실제 사용자 데이터로 카트 필요성 검증 가능 · 범위 제한으로 리스크 낮음 | · 제품이 "반쪽": 주문 확정 전 수정 불가 · 후속 카트 추가 시 `place_order` API 전면 재설계 (재작업 비용 2~3배) · 고객 경험 불만(키오스크에서 실수 롤백 불가) · 업계 표준 미달 |

## 최종 판정 (Team-Lead)

### 결정의 근거

**1. 기술 복잡도의 재평가**

devils-advocate의 초기 주장("2~3 스프린트, 대규모 리팩토링")은 엔터프라이즈 이커매스 가정을 키오스크에 오배치한 결과였다. tech-architect의 코드 기반 검증으로 다음이 명확해졌다:
- 기존 `Order` 애그리거트가 이미 카트 개념을 암묵적으로 포함 (PENDING 상태, add_item, remove_item, total_amount)
- `InMemoryOrderRepository` 단일 프로세스 환경에서 세션 병합·멀티 디바이스 동기화는 불필요
- 신규 비용은 **use case 분해 + CLI 바인딩** = 5~7일 수준

→ **기술 부담이 예상보다 낮음**. cart 도입의 주요 장벽이 제거됨.

**2. 비용 대칭성**

product-owner와 devils-advocate의 주장이 모두 일리 있었으나, 양쪽 모두 **비용 기준을 비대칭으로 설정**했다:
- product-owner: "순차 출시 재작업 비용(O) vs 동시 설계 비용(X)"만 계산
- devils-advocate: "카트 개발 비용(O) vs order 단독 출시 재작업 이자(X)"를 누락

**대칭적 계산**:
- **동시 출시 경로**: 1 스프린트 (선행 결정 포함) + 테스트 + 통합
- **순차 출시 경로**: order 2~3일 + 사용자 검증 대기 (2~4주) + 카트 개발 1 스프린트 + order API 재설계 3~5일

→ **총 비용에서 동시 출시가 우월**. 특히 time-to-market 측면에서.

**3. Shift의 분석 가치**

모든 4명 패널리스트가 shift를 기록했다는 것은 **토론이 정보 획득을 이루었다는 신호**:
- devils-advocate가 가장 큰 shift (3:1 구도 → 만장일치): 코드 검증과 도메인 특수성 인정
- ux-specialist의 shift (절대적 필수 → 조건부): 비즈니스 모델 검증 조건화로 의사결정 정교화
- product-owner·tech-architect의 shift (비용/일정 조정): 선행 결정 게이트 명시화로 리스크 가시화

→ **합의가 단순 다수결이 아니라 상호 학습 기반**.

### 최종 결정

**"Cart + Order 동시 출시"를 선택한다. 단, 다음 조건이 충족된 경우에만 착수한다.**

#### 선행 결정 게이트 (Prerequisite)

1. **비즈니스 모델 확인** (ux-specialist 제안):
   - Q: "이 키오스크에서 다품목 동시 구매가 핵심 플로우인가?"
   - A가 YES → cart 필수 포함 (본 ADR 결정 유효)
   - A가 NO → order + Buy Now 단독으로 재평가

2. **카트 = PENDING Order 매핑 확정** (tech-architect):
   - Order 애그리거트의 `PENDING` 상태를 "카트"로, `CONFIRMED` 이후를 "주문"으로 명시
   - 별도 Cart 애그리거트 신설 불필요 (기술 부담 경감)

3. **중복 아이템 병합 규칙 결정** (devils-advocate, product-owner 강조):
   - 현재 `Order.add_item`은 동일 메뉴 추가 시 예외 발생 (order.py:46-47)
   - 결정: "에러 유지" vs "수량 자동 증가"? → 카트 UX는 후자 요구
   - 이 선택이 도메인 불변식 변경으로 이어지므로 **명시적 합의 필수**

4. **가격 변동 정책** (tech-architect):
   - 고객이 카트에 담은 상품의 가격이 체크아웃 시점에 변경된 경우?
   - 옵션: 고객 공지 / 자동 반영 / 가격 잠금 → **비즈니스 결정**

#### MVP 스코프 (Strict)

카트 작업 4개만 포함:
1. `AddToCart`: 상품을 카트에 추가
2. `RemoveFromCart`: 상품을 카트에서 제거
3. `UpdateQuantity`: 카트 항목의 수량 변경
4. `ViewCart`: 카트 내용 + 총액 조회

**제외 (후속 이터레이션)**:
- 쿠폰/할인 적용
- 세션 영속화 (persistence)
- 재주문 이력
- 프로모션 로직
- 재고 예약

#### 추정 일정

**1 스프린트 (5~7일)**:
- 도메인 결정 (1~2일)
- use case 구현 + 테스트 (3~4일)
- CLI 바인딩 + 통합 테스트 (1~2일)

## 맥락 의존성 (Context Dependency)

**이 결정은 특정 프로젝트 상태에 기반한다:**
- **단일 프로세스 키오스크**: 세션 병합, 멀티 디바이스 동기화 불필요
- **기존 Order 도메인 존재**: `OrderRepository`, `OrderItem` 값 객체 이미 구현
- **사용자 인증 미포함**: 로그인 카트 vs 익명 카트 구분 불필요 (현재)

**재평가 트리거** (이 조건이 변경되면 ADR 재검토):
- 영속 카트 요구사항 (사용자가 저장된 카트 복구 원함)
- 멀티 채널 확장 (모바일 앱 추가)
- 사용자 인증 도입 (로그인 필수)
- 대량 주문 / B2B 견적 시나리오
