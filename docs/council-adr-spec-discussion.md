# Council ADR과 Spec 레이어 분리에 대한 논의

날짜: 2026-04-22
맥락: `/council` skill이 생성하는 ADR이 어느 수준까지 상세해야 하는가, 그리고 "신입 개발자도 구현 가능한 spec"은 언제/어떻게 만들어야 하는가에 대한 설계 논의.

---

## 질문 1. Council의 ADR은 기술적으로 완전히 정립된 상태여야 하나?

**결론: ADR과 구현 spec은 분리한다.**

| 문서 | 역할 | 수명 |
|---|---|---|
| ADR | "왜 이 길을 택했는가" — 대안, 트레이드오프, 제약, 결정의 맥락 | 길다 (리팩터링에도 살아남음) |
| Implementation spec | 파일 경로, 함수 시그니처, 테스트 케이스, 의존성 순서 | 짧다 (코드 변경과 함께 갱신) |

ADR에 구현 디테일을 넣으면 코드가 바뀔 때마다 ADR이 낡아버리고, 결정 근거가 구현 디테일에 묻혀 "이 결정을 뒤집어도 되나?"를 판단하기 어려워진다.

권장 흐름:

1. `/council` → ADR (결정 + 근거, 기술 골격은 포함하되 파일/함수 단위까지는 X)
2. `/mini-specify` → 요구사항 레벨 spec (ADR을 입력으로 받음)
3. `/taskify` → 시니어가 집어들 수 있는 task 단위 (현재 harness의 최종 산출물)

## 질문 2. 여러 전문가가 토론하는 Council 맥락에서 ADR은 어떤 형태여야 이상적인가?

**핵심 원칙: ADR은 토론의 전사(transcript)가 아니라 결론의 증류물이다.** 토론은 발산적이지만 ADR은 6개월 뒤 누군가 2분 안에 읽고 "아, 그래서 이 길로 갔구나"를 이해해야 한다.

### 이상적인 ADR 템플릿

```markdown
# ADR-YYYY-MM-DD: {결정 제목, 동사로}

## Status
Accepted | Superseded by ADR-XXXX | Deprecated

## Context
- 어떤 문제/제약이 있었는가 (3-5줄)
- 왜 지금 결정해야 했는가 (트리거)

## Decision
우리는 X를 한다. (단정문 1-3문장)

핵심 기술 골격:
- 주요 컴포넌트/경계
- 핵심 인터페이스나 데이터 흐름 (코드 레벨 X, 개념 레벨 O)

## Considered Alternatives
| 옵션 | 장점 | 버린 이유 |
|---|---|---|
| A. ... | ... | ... |
| B. ... | ... | ... |

## Consequences
- **긍정**: 무엇이 쉬워지나
- **부정**: 무엇을 포기했나, 어떤 부채가 남나
- **후속 작업**: 이 결정 이후 필요한 것 (→ spec/task로 넘어갈 거리)

## Panel Notes (선택)
- Security 관점: ...
- Performance 관점: ...
- DX 관점: ...
```

### 여러 전문가 토론을 ADR에 녹이는 팁

1. **패널 발언을 그대로 옮기지 않는다.** "보안 전문가가 X를 걱정했고, 성능 전문가가 Y를 반박했다"가 아니라, 그 걱정이 Decision이나 Consequences에 어떻게 반영됐는지만 남긴다.
2. **반대 의견은 Alternatives 표에 "버린 이유"로 살린다.** 토론에서 진 주장도 "왜 그게 틀렸다고 판단했는가"가 미래의 나에게 가장 값진 정보다.
3. **합의 안 된 부분은 `Open Questions` 섹션**에 명시한다. ADR이 모든 걸 결정할 필요는 없다.
4. **토론 원문이 아까우면 별도 파일로** (`.dev/council-logs/YYYY-MM-DD-{topic}.md`) 보관하고 ADR에서 링크만 건다. ADR 본문은 얇게 유지.

### 길이 감각

- 짧은 결정: 한 페이지
- 큰 아키텍처 결정: 2-3페이지가 상한
- 그보다 길어지면 ADR이 아니라 design doc이다 — 분리한다.

## 질문 3. 신입 개발자도 읽고 구현할 수 있을 정도의 최종 spec은 언제 만들어야 할까?

현재 mini-harness가 생성하는 `spec.json`의 step은 **"시니어가 읽고 구현"** 수준이지 **"신입이 읽고 구현"** 수준은 아니다.

### 현재 수준 예시 (run-20260421-050742-5284/spec/spec.json)

> "kiosk/domain/events/base.py에 DomainEvent frozen dataclass(event_id: UUID, occurred_at: datetime) 작성"

### 신입이 바로 구현하려면 더 필요한 것

- 함수/메서드 시그니처 전체 (반환 타입, 예외 타입 포함)
- 테스트 케이스 입출력 예시 (given/when/then, 최소 3개 — happy path + edge + error)
- 상태 전이 시 발생 예외 종류 ("가드 메서드"로만 기술되어 있음)
- 의존성 주입 경로 — 어디서 누가 인스턴스화하고 어디로 넘어가는지
- 실패 시 롤백/트랜잭션 경계

### 권장: 기본적으로 만들지 않는다

이유 세 가지:

1. **투자 대비 가치가 낮다.** junior-ready spec을 쓰는 노력이 종종 구현 자체보다 크다. 그 시간에 AI가 바로 구현하는 편이 빠르다 — 이게 지금 mini-harness가 택한 길이다.
2. **스펙이 코드보다 빨리 낡는다.** 시그니처까지 박아두면 리뷰 한 번에 전부 스펙도 고쳐야 한다.
3. **신입 온보딩 문제와 AI 구현 문제는 다른 문제다.** 둘을 같은 문서로 풀려고 하면 둘 다 어정쩡해진다.

### 만들어야 하는 예외 상황

| 상황 | 이유 |
|---|---|
| 외부 팀/계약자에게 위임 | 같은 저장소 맥락을 공유 못 하니 명시적 계약 필요 |
| 규제/보안 요구사항 있는 기능 | 구현 전에 감사받을 문서 필요 |
| 공개 API/라이브러리 경계 | 외부 사용자가 시그니처에 락인됨 — 설계를 먼저 확정 |

내부 feature 개발에서 신입이 붙는 경우라면, junior spec을 문서로 만들기보다 **시니어가 task 하나를 페어링으로 함께 구현**하는 쪽이 거의 항상 더 빠르고 학습 효과도 크다.

### 굳이 harness에 넣는다면 — 선택적 L5 레이어

지금 흐름이 `interview → council(ADR) → requirements → spec(task) → execute`라면, spec과 execute 사이에 선택적 **L5: detailed-design**을 둘 수 있다. 단, 기본값은 꺼두고 플래그로 활성화한다:

```
/taskify --detail=junior   # 시그니처·테스트케이스·예외까지 생성
/taskify                    # 현재 수준 (기본)
```

산출물은 `spec/task-{id}-design.md` 형태로 태스크별 별도 파일. task executor는 이 파일이 있으면 참조하고, 없으면 지금처럼 동작.

**Tradeoff**: 이걸 추가하면 "AI가 알아서 하면 되는 걸 왜 문서로?"라는 본질 문제가 남는다. 기본 입장은 **지금 구조를 유지**하고, 정말 외부 위임이 필요해질 때 그 특정 task만 수동으로 풀어 쓰는 것.

## 요약: 레이어별 책임

| 레이어 | 산출물 | 수준 | 누구를 위한 것 |
|---|---|---|---|
| L1 Interview | `interview.json` | 요구사항 명확화 Q&A | 설계자 |
| L2 Council | `ADR.md` | "왜"의 증류 (결정 + 근거) | 미래의 팀, 리팩터링 판단자 |
| L3 Requirements | `requirements.json` | "무엇"의 명세 | spec 생성기 |
| L4 Spec/Task | `spec.json` | 시니어/AI가 읽고 구현 가능한 수준 | task-executor, 시니어 |
| L5 Detailed Design (선택) | `task-{id}-design.md` | 시그니처·테스트케이스 | 외부 위임 시에만 |

**핵심 원칙**: 각 레이어는 하나의 독자(audience)만 가진다. 여러 독자를 겸하려는 문서는 대부분 실패한다.
